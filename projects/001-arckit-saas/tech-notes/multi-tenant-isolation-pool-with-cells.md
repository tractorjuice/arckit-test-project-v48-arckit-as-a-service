## Summary

The pool-with-cells pattern (sometimes called "shared-everything bounded by partitions") is the most cost-effective tenant isolation pattern for SaaS that needs SME-affordable per-tenant economics, defence in depth in case of isolation-control failure, and a clean degenerate-to-single-tenant mode for sovereign deployments. ArcKit as a Service adopts it per ADR-001 and materialises it on managed hyperscaler primitives (managed Postgres + S3-class object storage + managed Kubernetes per cell) per ADR-002 and ADR-006.

## Key Findings

- **Pool-only is too risky; silo-per-tenant is too expensive**. Pool-with-cells caps the blast radius of a pool defect at one cell (1,000 tenants) while preserving the per-tenant cost economics (Principle 1 / BR-001) of a shared platform.
- **tenant_id propagation is the platform's identity primitive**. Every storage row, every object key, every event-stream message, every cache entry, every log line, every span attribute carries `tenant_id`. CI runs an automated isolation test suite on every change that synthesises two tenants and asserts cross-tenant access fails.
- **Cell-fill discipline is the cost lever**. New tenants land in a current cell; cell capacity is capped (1,000 tenants initial); a new cell is provisioned automatically when ~75% full. This keeps per-cell DB-instance floor cost amortised across enough tenants to remain inside the BR-005 cross-subsidy unit economics.
- **Sovereign mode is a degenerate cell**. The same codebase runs in single-tenant configuration where `tenant_id` is fixed at deployment; no cell-partitioning is needed; isolation between projects, roles, and communities-of-interest is enforced by the same default-deny authorisation mechanisms.
- **Defence in depth is the floor**. Any single isolation mechanism failing must not be sufficient to permit cross-tenant access. ArcKit's stack: tenant_id middleware → application-layer authorisation default-deny → row-level security in Postgres (second-layer guard) → S3 bucket policies keyed off prefix → Kubernetes namespace + network policy at the runtime layer (per ADR-006) → cell-level network isolation.
- **NCSC Cloud Security Principle 3 ("Separation between users") explicitly addresses this** as a Cloud Security Principle. Evidence pack (BR-006) maps cell-level isolation to NCSC CSP 3 directly.

## Relevance to Projects

- **ArcKit as a Service Project 001 (managed SaaS)** — directly applied per ADR-001 / ADR-006.
- **ArcKit as a Service Project 002 (sovereign deployment)** — applied as a degenerate single-tenant cell with within-deployment isolation between projects and communities-of-interest.
