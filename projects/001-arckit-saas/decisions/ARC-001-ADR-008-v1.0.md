## 2. Context and Problem Statement

ADR-001 selected the pool model. Pools require an explicit fairness mechanism — without it, a single noisy tenant degrades every other tenant in the cell, breaking NFR-A-003 (graceful degradation) and the SME affordability commitment indirectly (because cells must then be over-provisioned). Quotas also realise the commercial tier differences (BR-001 free vs paid) and protect the per-tenant AI budget from ADR-004.

Constraints:

- **Tenant-isolating** — one tenant's quota consumption never affects another's headroom (NFR-A-003).
- **Operator-tunable per tenant** — admin console can override quotas for a tenant (FR-014).
- **Tenant-visible** — tenants see their quotas, current consumption, and recent denials in their UI.
- **Free vs paid differentiation** — quotas are the lever (Principle 1, BR-001/005).
- **Fail-degraded, not fail-down** — when over quota, the tenant gets a clear error (HTTP 429 with retry-after / quota-reset header), not a service crash.
- **Sovereign reuse** — quotas configurable to "unlimited" in single-tenant deployments (project 002).

## 4. Considered Options

### Option A — Token-Bucket Per Tenant Per Resource Class, Centralised Counter, Tier-Driven Defaults, Operator-Overridable, Tenant-Visible **(Recommended)**

**Description**: Quotas are enforced at the gateway (and, for AI, at the AIAdaptor — see ADR-004) using token-bucket algorithms per `(tenant_id, resource_class)` keyed counters. Resource classes include: `api_requests` (per minute), `ai_tokens` (per month), `storage_bytes` (cumulative), `export_requests` (per day), `concurrent_streams` (instantaneous). Defaults derive from tenant tier; operators (FR-014) can override per tenant. Tenants see consumption and reset times in their admin UI (FR-002). When a tenant exceeds a quota, the response is HTTP 429 with `X-RateLimit-*` and `Retry-After` headers; for AI budget exhaustion, a structured error explaining the budget and reset.

**Implementation Approach**:

- **Centralised counter** (e.g., Redis with Lua atomic ops) per cell.
- **Quota enforcement** at API gateway (synchronous) and at AIAdaptor (synchronous before provider call).
- **Tier table** versioned in config; changes are PRs; tenants notified of any decrease.
- **Operator override** logged in audit; reverts on tier change unless explicitly persistent.
- **Tenant UI** shows consumption, last denied request, and the tier table for context.
- **Sovereign mode**: all quotas default to "unlimited" with a configurable override (project 002 NFR-S-001 fixed envelope).

**Wardley Evolution Stage**: Commodity (token-bucket); Custom-built (the per-tenant tiered scheme on top).

**Pros (Good)**:

- ✅ **Per-tenant bulkhead** materialised; NFR-A-003 satisfied.
- ✅ **Predictable economics** — free tier cost capped per tenant.
- ✅ **Standardised error contract** (RFC 6585 429) — clients integrate cleanly.
- ✅ **Sovereign reuse** trivial via config override.

**Cons (Bad)**:

- ❌ **Quota tuning is a continuous activity** — too tight = false positives; too loose = noisy tenant impact. Mitigated by per-tenant override + telemetry-driven review.
- ❌ **Centralised counter is a hot path** — must be highly available; mitigated by per-cell Redis with HA.

**Cost Analysis**: low. Counter infrastructure is small per cell.

## 5. Decision Outcome

### Chosen Option

**Option A — Token-bucket per (tenant_id, resource_class), centralised counter per cell, tier-driven defaults, operator override with audit, tenant-visible consumption and standardised 429 error contract.**

### Y-Statement

> In the context of running a pool-isolated multi-tenant SaaS where one noisy tenant must not degrade others, while supporting tier-based commercial differentiation and the ADR-004 per-tenant AI budget,
> facing the conflict between strict isolation guarantees and the operational complexity of per-tenant tuning,
> we decided for **a token-bucket-per-(tenant, resource-class) model with centralised per-cell counters, tier-driven defaults, operator overrides with audit, and tenant-visible consumption with a standardised 429 contract**,
> to achieve **NFR-A-003 bulkhead, predictable free-tier economics, AI-budget integration, and trivial sovereign reuse via config override**,
> accepting **continuous quota-tuning effort and counter-infrastructure HA requirements**.
