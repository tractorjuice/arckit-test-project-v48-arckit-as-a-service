## 3. Context and Problem Statement

### 3.1 Problem Description

Tenants must be able to leave the service at any time with **everything** they have created — artefacts, version history, lineage metadata, audit log, attachments — in formats they can read, edit, diff, and re-import without buying ArcKit again. The format underwrites:

- **The exit promise (BR-007)** — without a credible exit, SME tenants face increasing switching cost over time, undermining Principle 1 (SME affordability).
- **UK GDPR Article 20 (right to data portability)** — tenants are controllers; the controller's right to portability means the format must be machine-readable and structured.
- **UK GDPR Article 17 (right to erasure)** — once a tenant has their export, the deletion path (FR-010) must be honoured; the export thus *enables* erasure rather than blocking it.
- **Sovereign-mode parity (Principle 21, project 002)** — the air-gapped sovereign deployment performs backups in this same format. One format, two routes; same code-path on both sides.
- **AI-grounding traceability (FR-004 + Principle 9)** — generated artefacts must carry full provenance; the export must preserve that lineage.
- **Round-trip integrity** — what comes out must go back in. A tenant exporting then re-importing into a fresh ArcKit instance (SaaS or sovereign) must end up with byte-for-byte the same artefacts and the same version history.

**Problem statement as a question**: What format(s) should ArcKit use to export every byte of a tenant's content and metadata, such that the format is human-readable, open, machine-parseable, supports byte-for-byte round-trip, preserves lineage, and is the same format used for sovereign-mode backups in project 002?

### 3.2 Why This Decision Is Needed

- **Business context**: BR-001 (SME affordability — credible exit reduces switching-cost risk), BR-004 (G-Cloud — exit plan is a listing requirement), BR-005 (cross-subsidy — exit promise must apply equally to free and paid tiers), BR-006 (UK policy evidence), BR-007 (portability and exit — direct).
- **Technical context**: FR-005 (versioning + audit trail must export), FR-006 (full-fidelity export — direct), FR-010 (offboarding + deletion), FR-012 (tenant audit log access — exportable as JSON), NFR-I-001 (open API standards), NFR-I-002 (data portability — direct), NFR-C-002 (audit retention — audit log must export), NFR-P-003 (bulk export latency — < 10 minutes for typical SME tenant).
- **Regulatory context**: UK GDPR Article 17 (erasure), Article 20 (portability), Article 28 (processor obligations); GDS Service Standard Point 12 (Use open standards), Point 9 (Create a secure service — signed export); Technology Code of Practice Point 8 (Make things open) + Point 12 (Open standards).

### 3.3 Supporting Links

- **Requirements**: BR-001/004/005/006/007; FR-005, FR-006, FR-010, FR-012; NFR-I-001/I-002, NFR-C-001/C-002, NFR-P-003.
- **Related ADRs**: ADR-001 (parent — exports are tenant-scoped; tenant_id propagation extends to the export pipeline), ADR-004 (parent — backup format and export format share serialisation), ADR-005 (parent — audit log derives from observability pipeline; same redaction policy applies on export), ADR-006 (parent — export job runs on the same runtime; signing happens via the same image-supply-chain (Cosign) used for OCI artefacts).
- **Cross-project**: project 002 BR-001 (single codebase — the export format is also the backup format), project 002 NFR-I-001 (open standards parity), project 002 FR-006 (within-deployment isolation — export is project-scoped in sovereign mode).

## 4. Decision Drivers (Forces)

### 4.1 Technical Drivers

- **Byte-for-byte round-trip on import**
  - Requirements: FR-006, NFR-I-002.
  - The export must be the canonical serialisation; CI must be able to export → wipe → import → diff = empty.

- **Human-readable, diff-able primary content**
  - Requirements: Principle 4 (open standards), Principle 15 (developer experience), SD-6/SD-7 (SME tenant goals).
  - SME architects expect to `git diff` and review artefacts without proprietary tooling.

- **Machine-parseable structure**
  - Requirements: NFR-I-001 (OpenAPI), NFR-I-002, FR-008.
  - Non-Markdown content (audit logs, lineage, structured fields) needs JSON / YAML, not free-form text.

- **Preservation of full lineage and version history**
  - Requirements: FR-005, Principle 9 (data quality, lineage, portability).
  - Every artefact version, every editor identity, every AI-grounding source must be in the export.

- **Tamper-evident export archive**
  - Requirements: NFR-SEC-006, NFR-C-002.
  - Cosign signature + SHA-256 manifest = chain of custody verifiable offline.

- **Sovereign-mode parity**
  - Requirements: Principle 21, project 002 BR-001.
  - The same archive format is used for sovereign backups and tenant migration in project 002 — one format, one code-path.

- **Performance envelope**
  - Requirements: NFR-P-003 (< 10 minutes for typical SME tenant; FR-006 acceptance criterion).
  - Format choice must allow streaming production and consumption.

### 4.2 Business Drivers

- **Credible exit promise — preserves SME unit economics**
  - Requirements: BR-001, BR-007.
  - Switching cost low ⇒ SME tenants stay because the platform is good, not because they're locked in.

- **G-Cloud exit-plan defensibility**
  - Requirements: BR-004.
  - G-Cloud requires a documented exit plan; this ADR is its centre.

- **Free-tier parity on export**
  - Requirements: BR-001 (no paywall for governance-critical features), FR-006 (export available on every tier including free).
  - Format must not be tier-gated.

- **Recognisable to UK Government assurance**
  - Requirements: BR-006.
  - Open formats (Markdown, JSON, YAML) are the dominant idiom across HMG digital programmes.

### 4.3 Regulatory & Compliance Drivers

- **UK GDPR Article 20 (Data portability)**: format must be structured, commonly used, and machine-readable.
- **UK GDPR Article 17 (Right to erasure)**: export enables erasure; deletion path (FR-010) must be honoured after export delivered.
- **UK GDPR Article 28 (Processor obligations)**: tenant is controller; processor must return personal data on contract termination.
- **GDS Service Standard Point 12 (Use open standards)**: directly served.
- **GDS Service Standard Point 9 (Create a secure service)**: signed export, tamper-evident, no PII leakage outside the tenant scope.
- **Technology Code of Practice Point 8 (Make things open)**: format published; exit documented.
- **Technology Code of Practice Point 12 (Make use of open standards)**: directly served.

## 5. Considered Options

### Option 1: Markdown Artefacts + JSON Manifest + YAML Lineage in a Signed Tarball (Recommended)

**Description**: Each artefact (Principle, Requirement, ADR, HLD, etc.) is serialised as a Markdown file (the canonical authoring format ArcKit already uses) in a directory layout that mirrors the project structure. A top-level `manifest.json` enumerates every file, its SHA-256, its artefact type, version, and tenant scope. A `lineage.yaml` per artefact captures full version history (who, when, what changed, AI-grounding sources where applicable) and the audit-log subset relevant to that artefact. The tenant audit log (FR-012) is exported as a single `audit.jsonl` (JSON Lines, one event per line, machine-streamable). Attachments and binary content are stored in a `blobs/` directory referenced by SHA-256 from the manifest. The whole tree is packed into a `tar.gz`, signed with Cosign (per ADR-006 image-supply-chain conventions, reused), and accompanied by an offline-verifiable detached signature and SHA-256 archive manifest.

The same format is used for **sovereign backups in project 002** — sovereign deployments produce these archives on a configurable cadence to a customer-supplied storage path.

**Implementation Approach**:

- **Directory layout in the archive**:

  ```text
  archive/
    manifest.json                   # top-level inventory + SHA-256 of every file
    audit.jsonl                     # tenant audit log (FR-012)
    projects/
      <project-id>/
        ARC-<id>-PRIN-v1.0.md       # canonical Markdown artefact
        ARC-<id>-PRIN-v1.0.lineage.yaml
        ARC-<id>-REQ-v1.0.md
        ARC-<id>-REQ-v1.0.lineage.yaml
        ...
    blobs/
      sha256/<hash>                 # binary attachments, content-addressed
    schema/                         # JSON Schemas for manifest + lineage formats
    README.md                       # human-readable export overview + import instructions
  ```

- **Manifest schema** (JSON Schema published; versioned with backwards-compatibility window per NFR-I-001):
  - `archive_version`, `tenant_id`, `exported_at`, `exporter_version`, `entries[]` with `path`, `sha256`, `artefact_type`, `version`, `superseded_by`, `created_at`, `last_modified_at`.
- **Lineage schema** (YAML; one file per artefact):
  - Version history: `versions[]` with `version`, `editor`, `timestamp`, `diff_summary`, `parent_version`.
  - AI-grounding (where FR-004 generated content): `ai_grounding[]` with `prompt_id`, `model`, `source_artefacts[]`, `redacted` (per ADR-005 PII redaction policy).
- **Audit log format**: JSON Lines for streaming; one event per line; tenant-scoped; redacted of operator-only fields per ADR-005.
- **Signing**: Cosign keyless signing via Sigstore transparency log (the same chain used for OCI images in ADR-006); detached signature included in archive root for offline verification.
- **Streaming production**: pipeline writes Markdown + manifest entries as it walks the tenant's artefact tree; archive composed in a temp volume; signed; uploaded to a tenant-accessible signed URL with 24-hour expiry.
- **Streaming consumption**: import pipeline (used by sovereign migration and by CI round-trip test) reads the manifest first, validates SHA-256, then ingests artefacts in dependency order.
- **Round-trip CI test**: every release runs a fixture tenant through `export → wipe target instance → import → diff`. Diff must be empty (modulo deterministic identifiers). This is a release blocker (FR-006 acceptance criterion + project 002 single-codebase commitment).
- **Schema versioning**: archive includes a `schema/` directory; new schema versions are additive within the 12-month backwards-compatibility window per NFR-I-001.

**Wardley Evolution Stage**: Commodity (Markdown, JSON, YAML, tar, Sigstore Cosign — all utility services / open standards).

#### Good (Pros)

- ✅ **Open formats — no proprietary tooling needed** — Markdown / JSON / YAML readable in any editor or `git diff`.
- ✅ **Round-trip provable in CI** — release blocker; format itself is the test.
- ✅ **Human-readable as primary content** — SD-6 (SME architect) can review artefacts post-export with no extra software.
- ✅ **Machine-parseable structure** — manifest + lineage YAML support tooling.
- ✅ **Sovereign-mode reuse** — same format is the project 002 backup format (Principle 21, project 002 BR-001).
- ✅ **Tamper-evident** — Cosign signature + SHA-256 manifest = offline-verifiable chain of custody.
- ✅ **Free-tier compatible** — formats cost nothing; export pipeline is the same regardless of tier (BR-001).
- ✅ **G-Cloud exit-plan defensibility** — direct match to BR-004 / BR-007 expectations.
- ✅ **UK GDPR Article 20 compliance** — structured, commonly used, machine-readable.
- ✅ **Low storage cost** — gzip on Markdown + JSON + YAML compresses well; manifest + lineage are small.
- ✅ **Reuses existing supply-chain primitives** — Cosign already in use per ADR-006; no new tools.

#### Bad (Cons)

- ❌ **Schema discipline must be maintained indefinitely** — every new artefact type and every new metadata field requires a schema update. Mitigated by JSON Schema + CI lint on the export.
- ❌ **Multi-format archive (Markdown + JSON + YAML + JSONL) is more shapes than a single-format archive** — but each shape fits the data type best (prose / inventory / structured-but-readable / streamable events). Mitigated by `README.md` and `schema/` in archive.
- ❌ **No native graph queries** — relationships between artefacts are expressed via IDs in YAML, not as RDF triples. Mitigated by per-artefact `lineage.yaml` plus `manifest.json` cross-references; full graph rebuildable on import.
- ❌ **Round-trip discipline burden in CI** — every artefact-affecting change must verify the round-trip. Mitigated by automated CI bench; this is a feature not a bug (forces format completeness).

### Option 2: JSON-LD Knowledge Graph (One Big JSON-LD Document Plus Attachments)

**Description**: Tenant content serialised as a single JSON-LD document that captures the artefact graph as RDF triples with a custom vocabulary. Attachments referenced by URI. The whole graph is one machine-parseable document.

**Implementation Approach**:

- One `tenant-graph.jsonld` document; `@context` defines the ArcKit ontology; every artefact is a node; every reference is a triple.
- Attachments in a `blobs/` directory.
- Signed bundle.

**Wardley Evolution Stage**: Product (well-understood pattern; less ubiquitous than Markdown for prose content).

#### Good (Pros)

- ✅ **Native graph queries on import** — rich semantic relationships; SPARQL-able.
- ✅ **One canonical document** — easier to validate against a single schema/ontology.
- ✅ **W3C standard (JSON-LD 1.1)** — open and well-defined.

#### Bad (Cons)

- ❌ **SME accessibility damaged** — JSON-LD is not the format an SME architect opens in `git diff`. Markdown of an ADR in JSON-LD is escaped-string content, not a readable artefact. Conflicts with SD-6 / SD-7 stakeholder goals and Principle 15 (developer experience).
- ❌ **Round-trip test more complex** — diffing two JSON-LD graphs requires graph isomorphism, not text diff.
- ❌ **No sovereign-mode advantage** — backups in JSON-LD are no easier to operate than tarballs; loses the "open the file in any editor" advantage that matters in incident response.
- ❌ **Custom vocabulary maintenance** — the ArcKit ontology becomes a long-lived contract requiring careful evolution.
- ❌ **Larger archive at small tenant sizes** — JSON-LD verbosity outweighs compression savings until many artefacts exist.
- ❌ **Schema-evolution burden equivalent** — same as Option 1 plus the ontology layer.

### Option 3: Vendor-Proprietary Archive (Custom Binary or Postgres `pg_dump`)

**Description**: Tenant's row-level Postgres data exported as a `pg_dump` archive (or a custom binary format), with attachments alongside.

**Implementation Approach**:

- Tenant rows extracted from the per-cell DB (per ADR-001 / ADR-004); `pg_dump` filtered by `tenant_id`.
- Attachments alongside; archive signed.

**Wardley Evolution Stage**: Commodity (`pg_dump`) but proprietary at the *semantic* layer (the ArcKit table schema is the contract).

#### Good (Pros)

- ✅ **Smallest implementation effort** — `pg_dump` is one command.
- ✅ **Database-native fidelity** — every column, every constraint, every index.

#### Bad (Cons)

- ❌ **Lock-in by another name** — a `pg_dump` of the ArcKit schema is unusable without an ArcKit instance; the SME tenant cannot open it in `git diff` or any editor; conflicts with BR-007 / Principle 4 / Principle 1.
- ❌ **Schema = the contract** — every internal table change becomes a tenant-visible breaking change; the platform's schema becomes externally-frozen.
- ❌ **Erasure (Article 17) harder to evidence** — restoring a `pg_dump` involves the same per-cell DB; cross-cell migration becomes an engineering project.
- ❌ **Sovereign migration brittle** — moving a tenant from SaaS to sovereign requires schema-equivalent Postgres on the receiving side; couples the runtime to the export.
- ❌ **GDS / TCoP open-standards posture compromised** — hard to defend at assessment.

## 6. Decision Outcome

### 6.1 Chosen Option

**Option 1 — Markdown artefacts + JSON manifest + YAML lineage in a signed tarball, with a CI-tested round-trip import. Same format used for sovereign-mode backups in project 002.**

### 6.2 Y-Statement

> **In the context of** running ArcKit as a Service as a multi-tenant SaaS for SMEs supplying UK government, where tenants must be able to leave at any time with everything they've created in formats they can read without buying ArcKit again — and where the same codebase must produce backups in a sovereign deployment (project 002) inside an air-gapped boundary,
> **facing** the conflict between schema-rich machine-parseability (one big graph or DB dump) and human-readable, diff-able, SME-accessible plain-text content,
> **we decided for** Markdown artefacts + JSON manifest + YAML lineage + JSONL audit log in a signed tarball, with published JSON Schemas, a round-trip CI test that blocks releases, and the same format reused as the sovereign-mode backup format,
> **to achieve** an open-standards exit promise, free-tier parity, UK GDPR Article 20 / 17 compliance, G-Cloud exit-plan defensibility, sovereign-mode parity at the data-format layer, and `git-diff`-readable artefacts for SME tenants,
> **accepting** schema-evolution discipline indefinitely, a multi-shape archive (Markdown + JSON + YAML + JSONL) rather than a single-format graph, and the obligation to make every release pass the round-trip test.

### 6.3 Justification

1. **Principle 4 (Open standards) — non-negotiable** — Markdown / JSON / YAML are the most open formats available for prose / inventory / structured-but-readable content respectively, all already ubiquitous in the HMG digital community.
2. **BR-007 (Exit promise) and Principle 1 (SME affordability)** — the format must be openable in any editor an SME architect already has. Option 2's JSON-LD graph and Option 3's `pg_dump` both fail this test.
3. **Principle 21 (Sovereign reuse)** — using the same archive format for sovereign-mode backups means one code-path, one round-trip test, and one operational runbook covers both routes.
4. **CI-tested round-trip = format completeness** — making round-trip a release blocker means the format itself is automatically tested every change. New artefact types must serialise *and deserialise* before they can ship.
5. **Tamper-evident provenance reuses ADR-006 infra** — Cosign signing already in use for OCI images; the export gets the same chain of custody for free.
6. **UK GDPR Article 20 and Article 17 served simultaneously** — structured, commonly used, machine-readable; export delivered → erasure path (FR-010) honoured with destruction certificate.

**Stakeholder consensus**: Service Owner + DPO + (PENDING) Lead Architect + project 002 liaison aligned on sovereign-format reuse as the deciding force.

**Dissenting view to record**: A view exists that JSON-LD would offer richer semantic queries and a single canonical document. Rejected because the SME-accessibility of plain Markdown is the dominating force for the target audience; semantic richness can be reconstructed from `manifest.json` + per-artefact `lineage.yaml` cross-references on import.

**Risk appetite**: Format completeness is dominant; schema-evolution burden is acceptable in exchange for SME-accessibility and sovereign-mode parity.

### 7.2 Negative Consequences (Accepted Trade-offs)

- ❌ **Schema-evolution discipline indefinitely** — every new artefact type and metadata field requires schema update + round-trip test update.
  - *Mitigation*: JSON Schema + CI lint; new fields additive within 12-month backwards-compatibility window per NFR-I-001.
- ❌ **Multi-shape archive (Markdown + JSON + YAML + JSONL)** — more shapes than a single-format graph.
  - *Mitigation*: `README.md` + `schema/` in archive; each shape fits the data type best.
- ❌ **No native graph queries on the export itself** — relationships are ID-cross-references, not RDF triples.
  - *Mitigation*: tooling can rebuild a graph from manifest + lineage on import; the on-disk shape is optimised for human review, not graph traversal.
- ❌ **Round-trip test is a release blocker** — slows release cadence when the format breaks.
  - *Mitigation*: this is a feature; it's the contractual guarantee made to tenants.

### 7.3 Neutral Consequences

- 🔄 **Schema versioning policy** documented (additive changes within 12 months; breaking changes require parallel-run + migration).
- 🔄 **Export staging area** sits in same UK region per ADR-002; encrypted at rest; signed URL with 24-hour expiry.
- 🔄 **Documentation in archive** — every export contains a `README.md` describing the format and pointing to the published JSON Schemas.
- 🔄 **Schema repository** — JSON Schemas published to a public location alongside the OpenAPI spec (NFR-I-001).
- 🔄 **Engineering training** — every engineer briefed on the round-trip discipline as part of onboarding; the round-trip test is "the format" for the team.

### Appendix C: Schema-Evolution Policy

- **Additive changes** (new optional fields, new artefact types): allowed in any minor release; importers must ignore unknown fields.
- **Backwards-incompatible changes**: require an `archive_version` bump; import path must support N and N-1 for at least 12 months (NFR-I-001 backward-compatibility window); deprecation notice in `manifest.json`.
- **Field removal**: deprecated for a full release cycle; removed only after 12 months and an ARB-approved supersession.
- **Schema repository**: published alongside the OpenAPI spec (NFR-I-001) and tagged per release.
