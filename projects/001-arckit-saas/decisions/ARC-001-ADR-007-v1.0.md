## 2. Context and Problem Statement

Tenant portability is a Principle 4 commitment, a BR-007 commercial promise, and an FR-006 functional requirement. It is also the single most important anti-lock-in evidence asset for SD-1 / SD-3 / SD-13 stakeholders. The export must:

- Be **complete** — every artefact, every version, every audit event the tenant can see, every binary attachment.
- Be **open-format** — Markdown for documents, JSON for structured metadata, OpenAPI for the API surface, machine-readable manifest, no proprietary container.
- Be **round-trip-safe** — re-importing into a self-hosted ArcKit (the open-source CLI / sovereign mode) yields the same workspace state.
- Be **byte-for-byte deterministic** — same export from same state produces identical bytes (modulo timestamp).
- Be **tenant-scoped** — exactly one tenant per export, no leakage.
- Be **available without vendor intervention** — UC-3 requires the export to be initiated and downloaded entirely self-service by the tenant.

This ADR records the export format and the round-trip guarantee.

## 4. Considered Options

### Option A — ZIP Archive of Markdown + JSON Manifest + Per-Document Sidecar JSON, with Round-Trip Test in CI **(Recommended)**

**Description**: The export is a ZIP archive whose layout mirrors the on-disk structure of the open-source ArcKit (which is the same shape sovereign mode uses). At the root: a `manifest.json` describing every file, content hash, schema version, generation timestamp, and audit-log digest. Per-document sidecar JSON files record metadata not naturally fitted to Markdown frontmatter (e.g., comment threads, version history, attachment references). All times normalised to UTC; all files lexically ordered in the manifest; whitespace and line-endings normalised. Re-importing the archive into a clean self-hosted ArcKit instance yields a workspace whose serialisation is byte-for-byte equal to the export (round-trip equality test in CI).

**Implementation Approach**:

- **Format**: ZIP container; deterministic ZIP (stable file order, fixed timestamps in metadata).
- **Layout**: same as sovereign-mode on-disk layout — `projects/<id>/...md`, `decisions/`, `external/`, etc.
- **Manifest**: `manifest.json` at root — version, tool version, tenant_id, timestamp, file list with SHA-256, audit-log digest.
- **Sidecars**: per-document `*.meta.json` for metadata not in frontmatter (comment threads, version diff history, AI provenance summaries).
- **Audit log**: separate `audit.jsonl` with hash chain anchored in manifest.
- **Attachments**: as files under `attachments/`; references in manifest with content type.
- **Schema version**: explicit; backward-compatible additions allowed within a major version.
- **CI round-trip test**: per release, generate fixture-tenant export, import into clean instance, re-export, assert byte equality (modulo controlled timestamp set).
- **Self-service export endpoint**: tenant admin initiates; download URL signed and short-lived; no operator intervention.
- **Sovereign reuse**: the same archive shape is what sovereign-mode reads natively (project 002 NFR-I-002).

**Wardley Evolution Stage**: Custom-built (the export logic + round-trip test); Commodity (the formats themselves: ZIP, JSON, Markdown).

**Pros (Good)**:

- ✅ **Deterministic byte output** — supports anti-lock-in evidence (verifiable equality).
- ✅ **Open formats only** — no proprietary archive; no per-tenant decryption tool needed.
- ✅ **Round-trip with sovereign mode** is a single test in CI; gives tenants confidence.
- ✅ **Self-service** — UC-3 met without operator intervention.
- ✅ **Audit-log integrity** preserved across export.

**Cons (Bad)**:

- ❌ **Determinism discipline** required — easy to introduce non-deterministic ordering inadvertently. Mitigated by CI byte-equality test.
- ❌ **Manifest schema must remain stable** — versioned with deprecation policy.
- ❌ **Large-tenant export** can be GB-scale; export is asynchronous with progress UI.

**Cost Analysis**: low engineering cost; modest object-store cost for staging large exports.

## 5. Decision Outcome

### Chosen Option

**Option A — Deterministic ZIP archive of Markdown + JSON manifest + per-document sidecars; round-trip equality test in CI; layout symmetric with sovereign-mode on-disk format.**

### Y-Statement

> In the context of providing tenant data portability that is verifiable, self-service, and round-trip-safe with the open-source / sovereign deployment of ArcKit,
> facing the conflict between proprietary container convenience and the Principle 4 open-standards / anti-lock-in commitment,
> we decided for **a deterministic ZIP archive of Markdown + JSON files matching the sovereign-mode on-disk layout, with a manifest, per-document sidecars, an integrity-chained audit log, and a CI round-trip equality test**,
> to achieve **verifiable anti-lock-in evidence, UC-3 self-service exit, and zero-friction reuse in sovereign mode**,
> accepting **engineering discipline to preserve determinism and a stable, versioned manifest schema**.

### Appendix B: Round-Trip Equality Procedure

1. Generate fixture tenant with deterministic content.
2. Export → archive A.
3. Import archive A into clean instance.
4. Re-export → archive B.
5. Assert SHA-256(A) == SHA-256(B), modulo a controlled timestamp difference field.
