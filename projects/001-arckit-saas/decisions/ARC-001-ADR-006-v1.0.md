## 2. Context and Problem Statement

ADR-001 fixes the cell-based pool isolation model. ADR-002 fixes the residency boundary and primitive shape. This ADR realises those decisions concretely: how cells materialise, what container runtime hosts the application, how ingress routes traffic, how IaC describes everything, and how the same shape collapses into a single-host deployment for project 002 (or, equivalently, a small managed Kubernetes cluster a customer chooses to run themselves).

Constraints:

- **Cells (ADR-001)** must be independent failure / blast-radius / DR domains.
- **Multi-AZ (ADR-002)** — workload spans ≥3 AZs in the primary region.
- **GitOps and reproducibility** (Principle 12; NFR-M-001/M-003) — every environment is described in code, every change reviewed.
- **Stateless services + managed stateful primitives** (ADR-001/002) — application stateless; state in DB / object store / KMS.
- **Cost discipline** (Principle 17) — workload right-sized; spot / preemptible where safe; per-cell cost tagged.
- **Sovereign reuse** (Principle 21; project 002) — same container images run in single-host or small-cluster mode in air-gapped environments.

## 4. Considered Options

### Option A — Managed Kubernetes (Per-Cell Cluster) on Chosen Hyperscaler with OCI Containers, IaC via OpenTofu/Terraform, GitOps via Argo/Flux **(Recommended)**

**Description**: Each cell is one managed Kubernetes cluster (e.g., AWS EKS, Azure AKS, GCP GKE — selected after `/arckit:research`). Cluster spans ≥3 AZs. Application services are stateless OCI containers. State sits outside the cluster in managed primitives chosen per ADR-002 (Postgres-compatible DB, S3-compatible storage, KMS, queue). Ingress via the chosen cloud's managed load balancer with WAF in front. IaC in OpenTofu/Terraform; application deployment via Helm with GitOps controller (Argo CD or Flux). Service mesh evaluated for mTLS / per-service authz; default to a lightweight CNI policy + vendor-managed mTLS where available; full mesh on opt-in.

**Implementation Approach**:

- **Per-cell cluster** with strict naming and IaC parameterisation (`cell_id`, `region`, `az_layout`, `tenant_capacity`).
- **Cluster sizing**: small node pool by default; scales up via cluster autoscaler; spot/preemptible for stateless workloads where eviction-tolerant.
- **Tenant_id propagation** at the application layer; cluster network policy denies cross-cell traffic by default.
- **Ingress**: managed L7 LB → WAF → ingress controller → service mesh (or sidecar) → service.
- **TLS**: cert-manager + ACME provider; private CAs for service-to-service.
- **IaC**: OpenTofu / Terraform modules per cell; module versioning; drift detection in CI.
- **GitOps**: cluster state in `git`; Argo CD or Flux reconciles; PR-driven changes; signed commits.
- **Right-sizing and FinOps tagging**: every resource tagged `cell_id`, `tier`, `purpose` for cost reports.
- **Sovereign reuse**: same OCI images run on:
  - Single-host docker-compose (project 002 small-site profile)
  - Small managed Kubernetes (project 002 mid-size profile)
  - Same Helm charts; configuration profile selects mode.

**Wardley Evolution Stage**: Commodity (Kubernetes, OCI containers, OpenTofu); Custom-built (cell-management automation built on top).

**Pros (Good)**:

- ✅ **Per-cell blast radius** materialised at infra layer.
- ✅ **OCI / Kubernetes portable** — sovereign reuse via same images, same charts.
- ✅ **GitOps** = audit trail of every infra change (NFR-C-002).
- ✅ **Multi-AZ native** in managed Kubernetes.
- ✅ **Right-sizing and spot** support SME affordability.

**Cons (Bad)**:

- ❌ **Per-cell cluster floor cost** — each cluster has a control-plane fee. Mitigated by cell-fill discipline and small initial node pool.
- ❌ **Kubernetes operational overhead** — cluster upgrades, node lifecycle. Mitigated by managed K8s + automated upgrade plan.
- ❌ **Service-mesh complexity** if adopted naively — defaulting to lightweight network policy + opt-in mesh.

**Cost Analysis**:

| Component | CAPEX | OPEX (annual) | TCO 3-year |
|-----------|-------|---------------|-----------|
| Managed K8s control plane (per cell) | TBD | low–medium | medium |
| Worker node pool (per cell) | TBD | medium (autoscaled) | medium |
| Ingress + WAF | TBD | low | low |
| GitOps tooling | low | low | low |

**GDS Service Standard Impact**: Points 9, 12 (open standards), 14 (reliable service).

### Option B — Serverless-First (Per-Cell Function-Based)

**Description**: Each cell is a collection of functions / serverless containers (e.g., Lambda + Fargate; Cloud Run; Azure Functions). No persistent cluster.

**Pros**:

- ✅ Lowest idle cost.
- ✅ No cluster lifecycle.

**Cons**:

- ❌ **Cold-start latency** affects NFR-P-001 interactive response budget.
- ❌ **Cross-provider portability lower** — function platforms diverge more than Kubernetes.
- ❌ **Sovereign reuse much harder** — air-gapped function platform is rare and bespoke.
- ❌ **Long-running streaming workloads** (AI generation streaming, FR-004) are not the natural fit.

**Verdict**: Not selected as the platform; serverless components used **inside** Option A where they fit (e.g., scheduled jobs, lightweight webhooks).

---

### Option C — Long-Running VMs (No Containers)

**Pros**: Simplest mental model.

**Cons**: Patching, scaling, packaging, sovereign reuse all worse. Not selected.

---

### Option D — Multi-Tenant in One Big Kubernetes Cluster (No Cell Boundary at Infra Layer)

**Cons**: Breaks cell blast-radius mandate from ADR-001. Not selected.

## 5. Decision Outcome

### Chosen Option

**Option A — managed Kubernetes per cell, OCI containers, IaC in OpenTofu/Terraform, GitOps via Argo/Flux, default lightweight network policy with opt-in service mesh, multi-AZ within cell, ingress via managed LB + WAF.**

### Y-Statement

> In the context of materialising the cell-based pool isolation model and the open-standards primitives at the runtime layer for a UK-resident multi-tenant SaaS that must also collapse onto single-host or small-cluster sovereign deployments,
> facing the conflict between managed-Kubernetes per-cell cost and the operational simplicity of one-cluster designs that violate cell isolation,
> we decided for **a managed Kubernetes cluster per cell on the chosen hyperscaler, OCI containers, IaC + GitOps reconciliation, multi-AZ, lightweight network policy with opt-in mesh, and ingress via managed LB + WAF**,
> to achieve **per-cell blast radius materialised at infrastructure level, sovereign-reusable images and charts, GitOps audit trail, and right-sized cost discipline**,
> accepting **per-cell cluster floor cost (mitigated by cell-fill discipline) and Kubernetes operational overhead (mitigated by managed offering)**.

## 6. Consequences

### Positive

- Cell isolation is a hard infrastructure boundary, not just an application convention.
- One container image runs in SaaS, customer-managed K8s, and air-gapped single-host modes.
- GitOps + IaC = full audit trail of every change (NFR-C-002).

**Measurable Outcomes**:

| Metric | Target | Source |
|--------|--------|--------|
| Cells = independent K8s clusters | 100% | ADR-001 |
| Cluster spans ≥ 3 AZs | 100% | NFR-A-001 |
| Infra change without GitOps PR | 0 | Principle 12 |
| Container image reuse (SaaS ↔ sovereign) | 100% (same digest) | Principle 21 |
| Right-sizing review (cost vs utilisation) | quarterly | Principle 17 |

### Negative (Accepted)

- Per-cell cluster floor cost; mitigated by cell-fill discipline.
- K8s upgrade cycle; mitigated by automated upgrade plan and pre-prod cell rehearsal.

### Neutral

- Cell-management tool runs the cell lifecycle (create/scale/decommission).
- Network policy default-deny; ingress controlled per service.
- Container security posture: image scanning in CI; runtime CVE feed; admission policy (signed images, no privilege escalation).

### Risks and Mitigations

| Risk | Likelihood | Impact | Mitigation | Owner |
|------|------------|--------|------------|-------|
| Cluster outage (control-plane) | LOW | HIGH | Managed K8s SLA; cross-cell DR test | SRE |
| Misconfigured network policy allows cross-cell | LOW | CRITICAL | CI policy lint; isolation test post-deploy | Security Lead |
| Image with vulnerability deployed | MEDIUM | HIGH | Scan-and-block in CI; runtime CVE feed; signed images | Security Lead |
| GitOps drift | LOW | MEDIUM | Drift detection; alert on deviation | SRE |
| Sovereign-mode regressions in shared chart | MEDIUM | MEDIUM | CI runs single-host smoke test on every release | Engineering |

## 7. Validation and Compliance

- **Post-deploy isolation test**: from cell A pod, attempt to reach cell B service — must fail.
