## 3. Context and Problem Statement

### 3.1 Problem Description

ArcKit must support three distinct identity populations: SME tenant users (UC-1 self-service onboarding, no enterprise IdP), enterprise tenant users (already operate Entra/Okta/Ping/etc. and require federated SSO), and vendor service operators (FR-014 admin console with phishing-resistant MFA). The same identity layer must be re-usable in project 002 sovereign deployments where the customer IdP is the only identity authority (project 002 FR-007).

**Problem statement as a question**: How do we authenticate three different populations under one identity layer that is SME-affordable, enterprise-federation-complete, operator-hardened, and portable to sovereign deployments?

### 3.2 Why This Decision Is Needed

- **Business context**: BR-001 (SME free tier — zero per-tenant integration cost), BR-005 (enterprise SSO without bespoke engineering), BR-006 (UK Government policy evidence — GOV.UK One Login adoption path).
- **Technical context**: NFR-SEC-001 (authentication, MFA), NFR-SEC-002 (tenant isolation — identity-anchored per ADR-001), NFR-SEC-003 (authorisation), NFR-SEC-007 (service-to-service auth), FR-007 (SSO), FR-014 (admin console), INT-001 (tenant IdP).
- **Regulatory context**: NCSC password / authentication guidance; NIST SP 800-63B AAL2 / AAL3; UK GDPR Article 32; GDS Service Standard Point 9.

### 3.3 Supporting Links

- **Requirements**: BR-001/005/006; FR-007, FR-014; INT-001; NFR-SEC-001/002/003/007, NFR-C-001/005/009.
- **Related ADRs**: ADR-001 (parent — claim-based tenant_id resolution), ADR-002 (parent — UK-resident IdP).
- **Cross-project**: project 002 ADR-003 (Sovereign identity — same OIDC/SAML adaptor).

## 4. Decision Drivers (Forces)

### 4.1 Technical Drivers

- **MFA mandatory for write access**
  - Requirements: NFR-SEC-001.
  - IdP must enforce MFA; vendor IdP must support TOTP / WebAuthn.

- **Phishing-resistant MFA for service operators**
  - Requirements: NFR-SEC-001.
  - WebAuthn / FIDO2 hardware-key support required.

- **Federated SSO via OIDC and SAML 2.0**
  - Requirements: INT-001, NFR-I-001.
  - Both protocols supported; OIDC primary.

- **Tenant_id resolved from identity claims**
  - Requirements: ADR-001, NFR-SEC-002.
  - Claim mapping documented; default-deny if missing.

- **Identity primitives portable to project 002**
  - Architecture principle: 21 (single codebase).
  - Code talks to a generic OIDC/SAML IdP; vendor-IdP-specific calls forbidden in app code.

### 4.2 Business Drivers

- **SME free tier viable without per-tenant IdP setup cost**
  - Requirements: BR-001.

- **Enterprise SSO without bespoke engineering per tenant**
  - Requirements: BR-005, FR-007.

## 5. Considered Options

### Option 1: Vendor-Hosted OIDC IdP for SMEs + Self-Service OIDC/SAML Federation + Hardware-Key Admin IdP (Recommended)

**Description**: Use a managed identity provider (e.g., AWS Cognito, Auth0, WorkOS, Microsoft Entra External ID, managed Keycloak) for SME tenants who do not bring their own IdP. The same product surface offers self-service SAML 2.0 / OIDC federation configuration for enterprise tenants and government buying authorities. Service operators authenticate via a separate, hardened admin IdP with mandatory hardware-key WebAuthn. All identity flows resolve a `tenant_id` claim that is propagated through the request pipeline per ADR-001.

**Implementation approach**:

- **SME tenants**: vendor-hosted IdP; signup with email verification; MFA required for write access (TOTP free, WebAuthn supported).
- **Enterprise tenants**: tenant admin configures OIDC or SAML 2.0 federation via self-service UI (FR-007); domain-claim verification (DNS TXT) prevents tenant-domain hijack.
- **GOV.UK One Login**: configured as an external OIDC IdP at the per-tenant level — a buying authority can route their users via One Login; not exclusive (the same tenant may also have direct accounts).
- **Service operators (FR-014)**: separate admin IdP; mandatory hardware-key (WebAuthn / FIDO2) MFA; step-up authentication required for tenant-data-impacting operations; session lifetime ≤ 8 h.
- **Claim mapping** documented and version-controlled; tenant_id claim required and validated at every gateway boundary; default-deny if absent.
- **Token policy**: access ≤ 15 min, refresh ≤ 24 h (configurable per tenant), refresh-token rotation with reuse detection.
- **Project 002 reuse**: integration sits behind a generic OIDC/SAML adaptor; project 002 plugs in customer IdP (project 002 FR-007) with no application-level changes.

**Wardley Evolution Stage**: Commodity / utility (managed IdP and OIDC/SAML federation are well-understood patterns).

#### Good (Pros)

- ✅ **SME tenants onboarded with zero per-tenant integration cost** — preserves UC-1 and SME affordability.
- ✅ **Enterprise tenants federate without vendor engineering** — FR-007 self-service.
- ✅ **GOV.UK One Login supported as an external IdP** — BR-006 evidence; SD-13 alignment.
- ✅ **Phishing-resistant MFA for operators** — addresses the highest-blast-radius accounts.
- ✅ **Generic OIDC/SAML adaptor portable to project 002** — Principle 21 satisfied.
- ✅ **Claim-based tenant_id resolution** integrates cleanly with ADR-001 isolation model.

#### Bad (Cons)

- ❌ **Vendor IdP becomes a sub-processor** — must be UK-resident or covered by IDTA / Adequacy (DPO inventory; ADR-002).
- ❌ **Federation-onboarding bugs are tenant-impacting** — mitigated by domain verification and integration test harness for major IdPs.
- ❌ **Token-lifetime / refresh design must be airtight** — long-lived tokens are the worst case in a multi-tenant system. Mitigated by short access-token TTL + refresh-rotation with reuse detection.

#### Cost Analysis

- **CAPEX**: Engineering for self-service federation UI; admin IdP standup; hardware-key issuance for operators (small fixed cost).
- **OPEX**: Managed IdP per-MAU pricing; admin IdP licence; operator-key replacement.
- **TCO (3-year)**: Low; per-tenant identity cost rounds to a few pence at scale.

#### GDS Service Standard Impact

| Point | Impact | Notes |
|-------|--------|-------|
| 5 (Make sure everyone can use the service) | Positive | Self-service onboarding |
| 9 (Create a secure service) | Positive | MFA, federation, hardware-key admin |
| 12 (Use open standards) | Positive | OIDC + SAML 2.0 |

### Option 2: Build Own IdP

**Description**: Vendor builds and operates its own credential store, MFA, and federation engine.

**Implementation approach**: Custom credential store, custom MFA flow, custom federation engine.

**Wardley Evolution Stage**: Custom-Built (anti-pattern — re-implementing commodity).

#### Good

- ✅ Maximum control over identity layer.

#### Bad

- ❌ **Identity is not a differentiating capability** — building it duplicates commodity work (Wardley anti-pattern).
- ❌ **Highest-risk surface to take in-house** at small team scale.
- ❌ **Slower path to phishing-resistant MFA** and to enterprise federation completeness.
- ❌ Unfavourable cost trade-off vs commodity managed offerings.

#### Cost Analysis

- **CAPEX**: Significant engineering; security audit; pen test scope expansion.
- **OPEX**: Continuous credential-store ops, MFA support, federation maintenance.
- **TCO (3-year)**: High; not justifiable.

---

### Option 3: Federated-Only (Require Tenant IdP for Every Tenant)

**Description**: Every tenant must bring their own IdP; no vendor-hosted identity at all.

**Wardley Evolution Stage**: Product (uses standard federation primitives only).

#### Good

- ✅ Vendor never holds credentials.
- ✅ Lowest sub-processor surface.

#### Bad

- ❌ **Catastrophic for SME tenants** who do not have an IdP — UC-1 self-service onboarding broken.
- ❌ Direct conflict with Principle 1 / BR-001.
- ❌ **Onboarding latency** for every tenant equals the IdP-integration latency.

#### Cost Analysis

- **CAPEX**: Federation harness only; no managed IdP.
- **OPEX**: Per-tenant onboarding effort skyrockets.
- **TCO (3-year)**: Low direct cost but high opportunity cost — SME tier essentially impossible.

---

### Option 4: Single Government IdP Only (GOV.UK One Login Exclusively)

**Description**: All users must authenticate via GOV.UK One Login.

**Wardley Evolution Stage**: Product (relying on a single external service).

#### Good

- ✅ Single trust anchor; UK Government identity narrative.
- ✅ Strong policy alignment.

#### Bad

- ❌ One Login does not currently address the SME-supplier population at the level required.
- ❌ Excludes large enterprise tenants who federate from Entra / Okta.
- ❌ Concentration on a single external dependency.
- ❌ One Login outage = service outage with no fallback.

#### Cost Analysis

- **CAPEX**: One Login integration only.
- **OPEX**: Low — but availability risk is concentrated.
- **TCO (3-year)**: Low cost, high constraint.

---

### Option 5: Do Nothing (Baseline)

**Description**: Defer identity decision; allow developers to wire whatever credential store they reach for.

#### Good

- ✅ No immediate decision cost.

#### Bad

- ❌ Breaches NFR-SEC-001 (MFA mandatory) and NFR-SEC-002 (tenant_id propagation).
- ❌ Cannot launch — buying authorities will not pilot.
- ❌ Catastrophic blast-radius risk.

**Verdict**: Not viable. Documented for completeness as the formal baseline comparator.

## 6. Decision Outcome

### 6.1 Chosen Option

**"Option 1: Vendor-Hosted OIDC IdP for SMEs + Self-Service OIDC/SAML 2.0 Federation for Enterprise/Government Tenants (incl. GOV.UK One Login as an External IdP) + Hardware-Key WebAuthn Admin IdP for Service Operators"**

### 6.2 Y-Statement (Structured Justification)

> **In the context of** supporting three distinct identity populations across an SME-affordable multi-tenant SaaS that must also serve government buying authorities and remain portable to a sovereign deployment route,
> **facing** the conflict between SME zero-friction onboarding and enterprise federation completeness — and the asymmetric blast radius of the service-operator population,
> **we decided for** a vendor-hosted IdP for SMEs, self-service OIDC/SAML federation for enterprise/government (including GOV.UK One Login as an external IdP option), and a separate hardened admin IdP with mandatory hardware-key MFA for service operators, all behind a generic OIDC/SAML adaptor that project 002 reuses,
> **to achieve** fast SME onboarding, federation completeness without per-tenant engineering, AAL3-equivalent operator authentication, and a single identity-shape that the sovereign route can plug a customer IdP into,
> **accepting** vendor IdP as a sub-processor (DPO inventory), federation-onboarding test surface, and short access-token TTL with refresh-rotation discipline.

### 6.3 Justification

Option 1 is the only choice that simultaneously satisfies the three population needs and Principle 21 (sovereign reuse via the same adaptor).

**Key reasons**:

1. **SME free-tier UX preserved**: vendor IdP = zero per-tenant integration cost; self-service signup measured in seconds, not days.
2. **Enterprise federation completeness without per-tenant engineering**: BR-005 cross-subsidy economics depend on enterprise tenants federating without vendor effort.
3. **Operator population hardened separately**: hardware-key WebAuthn for the asymmetric blast-radius accounts; step-up for tenant-data-impacting operations.
4. **Project 002 reuse via the same adaptor**: no fork; project 002 simply configures customer IdP via the same OIDC/SAML interface.
5. **GOV.UK One Login supported but not required**: BR-006 policy alignment without dependency-concentration risk.

**Stakeholder consensus**: Service Owner, Vendor Security Lead, DPO aligned. Pilot DDaT Security Architects (SD-3) consulted; AAL3-equivalent operator MFA satisfies their primary concern.

**Risk appetite**: Vendor-IdP sub-processor risk acceptable given UK residency / IDTA mitigations and annual DPA review.

## 7. Consequences

### 7.1 Positive Consequences

- ✅ **Self-service SME onboarding remains seconds-to-minutes** (UC-1).
- ✅ **Enterprise federation requires zero per-tenant engineering** — supports BR-005 cross-subsidy economics.
- ✅ **Hardware-key operator MFA closes the most asymmetric blast-radius account category**.
- ✅ **Claim-based tenant_id resolution** integrates ADR-001 without bespoke per-IdP code.
- ✅ **Sovereign reuse without code changes** — project 002 ADR-003.

**Measurable outcomes**:

- Service-operator accounts on hardware-key MFA: 100 % (NFR-SEC-001).
- Tenant write actions without MFA: 0 (NFR-SEC-001).
- Federation-onboarding self-service completion (without vendor engineering): ≥ 95 % (FR-007).
- Tenant_id claim missing → request denied: 100 % (CI-tested per ADR-001).
- Refresh-token reuse incidents detected: 100 % terminated session-family.

### 7.2 Negative Consequences (Accepted Trade-offs)

- ❌ **Sub-processor inventory expands by one** (vendor IdP) — DPO inventory.
- ❌ **Federation onboarding test surface is non-trivial** — IdP-specific quirks (Entra, Okta, Ping, ADFS, Keycloak, One Login) tested in CI integration harness.
- ❌ **Refresh-token reuse detection must be implemented carefully** — a known foot-gun.

**Mitigation strategies**:

- **Sub-processor**: annual DPA review; alternative IdP options pre-evaluated.
- **Test surface**: CI harness covers Entra, Okta, Keycloak, ADFS, GOV.UK One Login (sandbox), generic OIDC, generic SAML 2.0.
- **Refresh-token reuse**: rotation + reuse detection; bind refresh to client + IP class; reuse → session-family revocation.

### 7.3 Neutral Consequences (Changes Needed)

- 🔄 **Sub-processor inventory** updated (DPO).
- 🔄 **Per-tenant federation status** visible to tenant admin (FR-007).
- 🔄 **Operator hardware-key issuance** process documented in Operationalize.
- 🔄 **Token-lifetime policy** versioned and surfaced to tenants.

### 7.4 Risks and Mitigations

| Risk | Likelihood | Impact | Mitigation | Owner |
|------|------------|--------|------------|-------|
| Vendor IdP outage | LOW | HIGH | SLA review; cached last-known-good identity briefly; status page (FR-009) | SRE |
| Federation misconfiguration leaks tenant access | MEDIUM | HIGH | Domain verification (DNS TXT); per-tenant test harness; default-deny on missing claim | Security Lead |
| Refresh-token theft / replay | MEDIUM | HIGH | Rotation + reuse detection; bind refresh to client + IP class | Security Lead |
| Operator account compromise | LOW | CRITICAL | Hardware-key MFA; step-up; session limits; audit-log alerts | Security Lead |
| Vendor IdP DPA term changes | LOW | MEDIUM | Annual DPA review; alternative IdP options pre-evaluated | DPO |
