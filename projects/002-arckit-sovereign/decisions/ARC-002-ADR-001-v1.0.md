## 2. Context and Problem Statement

Project 002 delivers ArcKit as a single-codebase deployable into customer-controlled environments — including disconnected MOD sites and other sensitive infrastructure. The release-artefact bundle must:

- Be **the same OCI images** built by the SaaS pipeline (Principle 21; project 002 BR-001) — no fork.
- Be **cryptographically signed** by a vendor key whose custody satisfies Customer Accreditors (SD-1) and Vendor Security Lead (SD-10).
- Carry a **Software Bill of Materials (SBOM)** in a recognised format (CycloneDX or SPDX), and verifiable provenance attestations (SLSA L3+ targeted).
- Be **transferable into air-gapped environments** via physical media or one-way diodes — without dependency on outbound network calls during install (project 002 NFR-SEC-004).
- Carry a **release manifest** (DR-007) describing every component, version, hash, signature, and security-relevant change.
- Support **delta / patch bundles** for the LTS line (project 002 BR-005, NFR-C-005, FR-014) — see ADR-002 for the update mechanism.
- Be **reproducible** so that a customer can independently rebuild from source and verify identity to the bundle they received.

This ADR records the packaging, signing, and distribution model.

## 4. Considered Options

### Option A — Single OCI-Image Bundle, Detached Sigstore Signature, CycloneDX SBOM, SLSA L3+ Provenance, Reproducible Build, Customer-Verifiable Manifest **(Recommended)**

**Description**: Each sovereign release builds a single archive (`arckit-sovereign-<version>.tar`) containing:

- All OCI images required to run the sovereign profile (same images as SaaS — Principle 21).
- Helm chart(s) for each supported deployment profile (single-host docker-compose, k3s, customer-K8s).
- IaC templates (OpenTofu modules) for known customer environments (optional; opt-in).
- A `release-manifest.json` (DR-007) listing every artefact, SHA-256, version, license, and security-relevant change since the previous release.
- A `sbom.cyclonedx.json` covering all images and host-side components.
- A `provenance/` directory with SLSA L3+ attestations.
- A `signatures/` directory with detached cosign signatures over the manifest, the SBOM, and each image.
- An `install/` directory with the air-gap install runbook (project 002 FR-001, RB-01 in `/arckit:operationalize`).
- An `INTEGRITY.md` describing the customer-side verification procedure.

**Implementation Approach**:

- **Build** in CI (DevOps strategy §3) on a hardened, pinned-base-image runner; provenance attached.
- **Sign** with a cosign key whose **private half is held in vendor HSM** (custody policy documented; SD-10 ratified). Public half published openly and pinned in customer trust store at first install.
- **Verify**: customer runs `cosign verify` (or air-gap equivalent) against the public key + manifest. Customer may also rebuild from source and compare image digests (reproducible build is a target, not a hard guarantee at v1; aspiration tracked).
- **Distribution**:
  - Default: encrypted physical media (USB / portable disk) for air-gapped customers, couriered with chain-of-custody.
  - Alternative: secure file transfer to customer's controlled exchange / cross-domain solution.
  - Never: public unauthenticated download for sovereign artefacts (separate from the open-source repo).
- **Patch bundles** (LTS line — BR-005, NFR-C-005, FR-014, ADR-002): same archive shape; smaller; signed by same key; manifest references the previous baseline.

**Wardley Evolution Stage**: Custom-built (the bundling pipeline) atop Commodity primitives (OCI, cosign, CycloneDX, SLSA).

**Pros (Good)**:

- ✅ **Same images as SaaS** — Principle 21 satisfied; no engineering fork.
- ✅ **Customer-verifiable provenance** — supports accreditation (SD-1).
- ✅ **No outbound install dependency** — air-gap safe (NFR-SEC-004).
- ✅ **Open-standards signing and SBOM** — cosign / CycloneDX / SLSA all open.
- ✅ **Patch bundle compatibility** — same shape simplifies LTS operations (SD-13).

**Cons (Bad)**:

- ❌ **Bundle size large** — full OCI image set plus Helm + IaC; mitigated by physical-media defaults and delta patches.
- ❌ **Vendor signing-key custody is critical** — compromise would undermine sovereign trust. Mitigated by HSM custody, dual control, and key-rotation runbook (RB-09).
- ❌ **Reproducible build is hard** — pursued aspirationally; customers can compare image digests + manifest as the practical control.

**Cost Analysis**:

| Component | CAPEX | OPEX (annual) | TCO 3-year |
|-----------|-------|---------------|-----------|
| HSM for signing key custody | medium (one-off) | low | low |
| CI extension for sovereign bundle build | low | low | low |
| Physical-media distribution logistics | low | low–medium | low–medium |
| LTS patch backport pipeline | medium (one-off) | low | low |

**MOD / NCSC Alignment**:

- MOD Secure by Design: provenance, SBOM, signing — primary evidence (project 002 NFR-SEC-001).
- NCSC supply-chain guidance: covered.
- JSP 440 / JSP 604: alignment via packaging discipline + customer-side verification.

### Option C — Online Update Mechanism Only

**Cons**: Breaks NFR-SEC-004 (no outbound calls). Not viable for air-gapped.

---

### Option D — Source Tarball + Customer Builds On-Site

**Pros**: Maximum customer control.

**Cons**: Build environment, dependency mirror, and reproducibility burden falls on customer; defeats the "fast install" objective and is unrealistic at most sites.

**Verdict**: Available as an opt-in path for customers who demand it (it's the open-source ArcKit), but not the **default** distribution.

## 5. Decision Outcome

### Chosen Option

**Option A — single OCI-image bundle, detached cosign signature with vendor HSM custody, CycloneDX SBOM, SLSA L3+ provenance, reproducible-build target, customer-verifiable manifest, physical-media-by-default distribution.**

### Y-Statement

> In the context of distributing a sovereign deployment of ArcKit into air-gapped MOD and sensitive-site environments while preserving Principle 21 single-codebase discipline,
> facing the conflict between vendor packaging convenience and customer-accreditor evidence demands,
> we decided for **a single OCI-image bundle with detached cosign signing (HSM-custodied), CycloneDX SBOM, SLSA L3+ provenance, customer-verifiable manifest, and physical-media-default distribution**,
> to achieve **customer-verifiable supply-chain integrity, air-gap-safe install, single codebase reuse from SaaS CI, and an LTS patch path that uses the same signing chain**,
> accepting **large bundle size (mitigated by physical media + delta patches), vendor signing-key criticality (mitigated by HSM + dual control + rotation runbook), and reproducible-build aspiration (not yet a hard guarantee)**.

### Appendix B: Customer-Side Verification (Concept)

```bash
# 1. Pin the vendor public key (one-time at first install)
cp arckit-vendor.pub /etc/arckit/trust/

# 2. Verify the bundle manifest signature
cosign verify-blob --key /etc/arckit/trust/arckit-vendor.pub \
   --signature signatures/manifest.sig release-manifest.json

# 3. Verify SBOM signature
cosign verify-blob --key /etc/arckit/trust/arckit-vendor.pub \
   --signature signatures/sbom.sig sbom.cyclonedx.json

# 4. Verify each image signature against image digests in manifest
for img in $(jq -r '.images[].digest' release-manifest.json); do
   cosign verify --key /etc/arckit/trust/arckit-vendor.pub <local-image-ref>@${img}
done

# 5. (Optional) Verify SLSA provenance attestations
```
