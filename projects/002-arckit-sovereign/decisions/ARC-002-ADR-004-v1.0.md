## 1. Context and Problem Statement

Sovereign deployments **must not** make outbound calls to vendor AI providers (NFR-SEC-004). At the same time, the sovereign route should still offer AI assistance **where the customer has approved a model**. Constraints:

- The same AIAdaptor from project 001 ADR-004 (Principle 21).
- AI **optional** — if no provider configured, AI degrades gracefully (NFR-A-003); manual authoring (project 002 FR-008) still works.
- Customer-approved providers may be:
  - On-prem GPU-served open-weight models.
  - Customer-controlled inference platform (e.g., MODNet-resident).
  - Customer-approved external endpoint via a permitted egress path (rare; subject to customer policy).
- Tenant_id propagation (single-tenant in sovereign) — same enforcement.
- Audit log of every generation (project 002 FR-010).

## 3. Considered Options

### Option A — Pluggable AIAdaptor Implementations (None / Local Open-Weight / Customer-Approved Endpoint), AI Off by Default, Generation Audit Always On **(Recommended)**

**Description**: AIAdaptor ships with three named implementations:

1. `none` (default) — AI features disabled; UI shows "AI generation not configured".
2. `local-openweight` — wraps a customer-deployed inference server (e.g., vLLM / TGI / Ollama) reachable inside the deployment boundary.
3. `customer-approved-endpoint` — wraps an HTTPS endpoint the customer has approved and is reachable per their network policy.

The provider is selected via Helm values at install (project 002 FR-005). Generation audit log (project 002 FR-010) is **always on** — regardless of which provider is configured. No outbound traffic except what the chosen provider implementation explicitly makes; default deployment is fully air-gap-compliant.

**Implementation Approach**:

- AIAdaptor interface unchanged from project 001.
- Per-provider implementation in `providers/`.
- Provider selection at install via `aiadaptor.provider` Helm value.
- Customer-approved endpoint config: URL, signing/auth, model name; documented INTEGRITY procedure.
- Audit log entry for every generation: model, tokens, latency, prompt-shape (template id), human-review-required flag.

**Pros**:

- ✅ Same adaptor; no code fork.
- ✅ Default safe (no AI; no outbound).
- ✅ Customer chooses model and infrastructure.
- ✅ Generation audit always on — provenance for accreditor (SD-1).

**Cons**:

- ❌ On-prem GPU operation is the customer's burden.
- ❌ Open-weight model capability may be lower than frontier proprietary models — communicated to customer.

---

### Option B — Bundle Open-Weight Model With Sovereign Release

**Cons**: Bundle size catastrophic; model versioning churn; customer-controlled model selection is lost. Rejected.

---

### Option C — No AI in Sovereign

**Cons**: Removes ArcKit's main differentiator; sovereign customers asking for AI cannot have it. Rejected; "no AI" is a configuration of Option A, not a separate architecture.

---

## 4. Decision Outcome

**Option A — Pluggable AIAdaptor; AI off by default; customer-supplied provider; generation audit always on.**

### Y-Statement

> In the context of offering AI assistance in air-gapped sovereign deployments without breaching NFR-SEC-004,
> facing the conflict between AI capability and the no-outbound-network mandate,
> we decided for **a pluggable AIAdaptor with three implementations (none / local open-weight / customer-approved endpoint), AI off by default, and generation audit always on**,
> to achieve **single-codebase reuse, customer-controlled AI choice, air-gap safety by default, and consistent provenance evidence**,
> accepting **on-prem GPU operational burden on the customer and possibly lower model capability**.

## 5. Consequences

### Positive

- Default sovereign deployment is air-gap safe and AI-disabled — simplest accreditation path.
- Customer can opt in when they have approved a provider.
- Same adaptor + audit pattern as SaaS.

**Measurable Outcomes**:

| Metric | Target | Source |
|--------|--------|--------|
| Default deployment outbound calls | 0 | NFR-SEC-004 |
| Generation audit log entries on AI use | 100% | FR-010 |
| AIAdaptor swappable without app code change | 100% | Principle 4 |

### Negative (Accepted)

- Customer GPU operation burden when local-openweight chosen.
- Capability gap vs frontier proprietary models for some workloads.

### Risks

| Risk | Mitigation | Owner |
|------|-----------|-------|
| Customer-approved endpoint exfiltrates prompt content | Customer-policy-controlled; vendor warns; audit logged | Customer |
| Local-openweight inference outage | Manual authoring fallback (FR-008) | Customer Operator |
| Model bias in chosen open-weight model | Customer-side evaluation; vendor publishes evaluation harness | Customer + Vendor |
