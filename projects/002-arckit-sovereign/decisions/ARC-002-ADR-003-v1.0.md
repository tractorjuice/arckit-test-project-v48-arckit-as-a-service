## 1. Context and Problem Statement

In sovereign deployments, **the vendor must not hold or operate any identity infrastructure** for the customer. Identity is provided by the customer's existing IdP (AD / Entra / MODNet identity / customer-operated Keycloak / etc.).

Constraints:

- **No outbound network calls** (NFR-SEC-004) — federation must be local to the deployment boundary.
- **Cleared-personnel-only access where mandated** (NFR-SEC-007) — IdP attestation of clearance level.
- **Reuse of project 001 ADR-003 OIDC/SAML adaptor** (Principle 21).
- **No vendor IdP fallback** — if the customer IdP is offline, the sovereign deployment is offline; no SaaS-to-customer leakage path possible by design.

## 3. Considered Options

### Option A — Customer IdP Only via OIDC/SAML 2.0 Adaptor with Cleared-Personnel Claim Mapping **(Recommended)**

**Description**: The same OIDC/SAML adaptor from project 001 ADR-003, configured at install time to talk to the customer IdP. Claim mapping is configurable: tenant_id (single-tenant in sovereign mode), role, optional `clearance_level` claim. If `clearance_level` is configured as required and the claim is missing, default-deny.

**Implementation Approach**:

- IdP endpoint URL, signing-key trust, claim mapping configured via Helm values at install (project 002 FR-005).
- Secrets / private keys live in customer KMS (project 002 INT-007).
- All IdP traffic stays inside the customer boundary (no outbound; NFR-SEC-004).

**Pros**:

- ✅ Reuses project 001 adaptor — single codebase.
- ✅ No vendor IdP — no surface area for vendor-side identity attack.
- ✅ Cleared-personnel claim respected.
- ✅ Customer keeps full identity control.

**Cons**:

- ❌ Per-customer integration effort at install time — mitigated by documented runbook + Helm values.
- ❌ If customer IdP is offline, sovereign deployment is offline — accepted; this is the correct behaviour for a customer-controlled environment.

---

### Option B — Vendor-Hosted IdP Embedded in Bundle

**Cons**: Duplicates customer-side identity authority; conflicts with the whole sovereign value proposition; rejected.

---

### Option C — No Federation, Local Username/Password

**Cons**: Breaks customer identity governance; not compliant with NCSC password / authentication guidance; rejected except as a last-resort lab-only option, documented but not supported.

---

## 4. Decision Outcome

**Option A — customer IdP only via OIDC/SAML 2.0 adaptor with cleared-personnel claim mapping.**

---

## 5. Consequences

### Positive

- Customer retains full identity authority.
- Same adaptor as SaaS — single codebase.
- Cleared-personnel access enforceable.

### Risks

| Risk | Mitigation | Owner |
|------|-----------|-------|
| IdP misconfiguration locks operators out | Break-glass local emergency account documented; activation logged | Customer Operator |
| Claim mapping omitted | Install-time validation; default-deny on missing required claim | Sovereign Delivery Lead |
| Customer IdP outage | Accepted: no vendor identity fallback by design | Customer |
