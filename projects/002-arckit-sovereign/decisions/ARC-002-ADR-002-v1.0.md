## 2. Context and Problem Statement

The sovereign LTS line (BR-005, NFR-C-005, FR-014) commits to delivering security patches and feature releases to sites that have **no outbound network**. Patch delivery must:

- Use the **same signing chain** as ADR-001.
- Be **forward compatible**: a site one version behind can apply patches without lifting baseline first.
- Provide a **roll-back path** (FR-002): a patch can be reverted without data loss within a published window.
- Carry a **compatibility manifest**: which baseline version this patch applies to; what database migrations it includes; what runbook the operator must follow.
- Be **operator-friendly**: a single command (or two-command) install path.
- Fit the **physical-media** distribution channel from ADR-001.

## 4. Considered Options

### Option A — Delta Patch Bundle With Compatibility Manifest, Single-Command Apply, Reversible Migrations Within Window **(Recommended)**

**Description**: Each patch is a **delta bundle** (`arckit-patch-<from>-<to>.tar`) containing only changed images, changed configuration, changed Helm values, DB migration up/down scripts, and a **compatibility manifest** describing supported baseline versions, migration reversibility window, and runbook references. Operator runs a documented single command (`arckit-apply patch-bundle.tar`) that:

1. Verifies signature (ADR-001).
2. Verifies baseline-compatibility.
3. Performs migration (with backup snapshot).
4. Performs cell-by-cell (or single-host) image swap with bake/health check.
5. Records `applied-patches.json` log.

Roll-back via `arckit-rollback <patch-id>` reverses migration (within window) and restores prior image set.

**Implementation Approach**:

- **Build**: same CI pipeline; bundle-diff utility produces patch artefact.
- **Migration discipline**: forward + reverse migration scripts; CI tests both paths.
- **Window**: roll-back supported until next forward patch is applied OR until 90 days, whichever sooner; documented per release.
- **Runbook**: RB-13 (hotfix) and RB-08 (cell upgrade) cover; sovereign-specific instantiation in `install/upgrade.md`.

**Pros**:

- ✅ Small bundle (delta) — fits physical media easily.
- ✅ Same signing chain — accreditor evidence consistent.
- ✅ Roll-back path closes the operator's biggest objection.
- ✅ Simple operator UX.

**Cons**:

- ❌ Migration discipline is non-trivial (forward + reverse + tested).
- ❌ Roll-back window must be communicated; missing it requires baseline reinstall.

---

### Option B — Full Bundle Re-Install Per Patch

**Cons**: Larger bundles; operator perceives every patch as a re-install — accreditation churn.

---

### Option C — Live Update Service (Outbound Pull)

**Cons**: Breaks NFR-SEC-004. Not viable for air-gapped sites; can be optionally supported for non-air-gapped sovereign customers (project 002 INT-009 vendor remote support channel) but not the default.

## 5. Decision Outcome

**Option A** — delta patch bundle, single-command apply, reversible migrations within published window, same signing chain as baseline.

### Y-Statement

> In the context of delivering LTS patches to air-gapped sovereign deployments without engineering bifurcation,
> facing the conflict between patch frequency and the operational cost of full re-installs,
> we decided for **delta patch bundles with compatibility manifest, single-command apply, reversible migrations within window, and the same signing chain as the baseline**,
> to achieve **fast, defensible patch delivery into air-gapped sites with a documented roll-back path**,
> accepting **migration discipline burden and the need to communicate the roll-back window clearly**.

## 6. Consequences

### Positive

- LTS patch SLA achievable.
- Roll-back gives operators confidence.
- Same signing chain — accreditation re-uses ADR-001 evidence.

**Measurable Outcomes**:

| Metric | Target | Source |
|--------|--------|--------|
| Patch bundle signature verifies | 100% | ADR-001 |
| Migration reversibility tested in CI per patch | 100% | This ADR |
| Operator single-command apply success | ≥ 95% | UX |
| Patch SLA met (Critical: ≤ 7 d; High: ≤ 30 d; Medium: ≤ 90 d) | ≥ 95% | NFR-SEC-008 |

### Negative (Accepted)

- Migration discipline ongoing.
- Roll-back window must be honoured operationally.

### Risks

| Risk | Mitigation | Owner |
|------|-----------|-------|
| Patch breaks customer baseline | Compatibility manifest + CI multi-baseline test | LTS Engineering Lead |
| Migration fails mid-flight | Backup snapshot pre-migration; clear failure → roll-back | SRE / Operator |
| Patch signing key drift | Rotation runbook (RB-09); key continuity policy | Security Lead |
